AIS performance baseline (first-order)
======================================
Date 2026-06-09. Build: c/ais -O2 (make). One desktop core, /tmp (local disk).
Reproduce: python3 tests/perf/gen.py /tmp/ais_1m 1000000
           ais -f /tmp/ais_1m -y compact      (builds idx/ from store)
           then the timings below. Generator seed 20260606 (deterministic).

Dataset
  records   1,000,000     (~ a heavy human lifetime: dozens/day x 365 x ~70 yr)
  store     85 MB, 1,000,000 lines  "id|keys|value"
  keys      4,000 distinct          (human active vocab ~10-20K => key-space BOUNDED)
  postings  1.8M total; hottest key = 270,360 ids; median key = 69 ids
  meta      1,000,000 lines  "id|epoch"  spanning 1996..2026

Timings (wall, one core)
  compact  (build whole index)        7.9  s    one streaming pass, 4000 lists
  find SUBSTR (full scan)             1.64 s    875,688 hits    -- grep-class
  dump (full scan)                    1.84 s    1,000,000 lines -- cat-class
  stats (streaming, ids only)         0.04 s
  put (single, at n=1M)               0.05 s    one O(n) idempotent store scan
  timeline: awk date-range on meta    0.37 s    33,520 records in ~2007
  get rare key (19 ids)               1.02 s    <-- O(matches x store)
  get hot key (270k ids)              hours     <-- killed; O(matches x store)
  generate (Python, NOT C)            132  s    the generator is the slow part

Reading
  - single-pass reads (find/dump/stats/timeline) are I/O-bound = grep/cat class. GOOD.
  - put is 50 ms interactively (a few/day = invisible). O(n^2) only on BULK import.
  - GET is the bottleneck: ais_record re-scans the store per matched id, so
    get = O(matches x store). 19 results = 1 s; a common key = minutes/hours.

Root cause (get AND bulk-put)
  No way to find a record by id except scanning the store.

Fix direction (keeps plain text, no hashing, all streaming)
  1. id->byte-offset index, built at compact (store grouped by id, contiguous reads)
     => ais_record = seek; get = O(matches). Sub-ms small gets, ~0.1 s hot key.
  2. put: O(1) append + batch dedup (sort-based), not a per-put O(n) scan.
  Net: every op becomes a seek or one streaming pass = find/rsync class.

Watch
  Re-run after adding features. A constant-factor slowdown is fine; flag any op
  that degrades by an ORDER OF MAGNITUDE and look there first.
