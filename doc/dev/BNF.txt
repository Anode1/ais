============================
AIS index storage grammar
============================

The formal grammar of the on-disk index format: the contract an automated writer
(a script, the GUI, the web/SQLite layer, an agent) must produce, and that a
verifier can check the code against in both directions throughout development.
See LAYOUT.md (same directory) for the directory map and algorithms; this file
is the format.

Grammar below is BNF-like, where terminals are delimited by " " (double-quotes),
non-terminals by < >, and simple regular expressions like character grouping [ ],
repetitions *, +, {N,M} are used. [:blank:], [:graph:], [:alpha:], [:punct:],
[:digit:] are standard POSIX character classes. ( ) denotes grouping, ? ? is a
verbal explanation of a character class. "\n" is LF.


---- store : the source of truth (append-only) -------------------------------
INDEX/store. One record per line. A v3 line is <id>|<ts>|<keys>|<value>; a
legacy v1 line (no <ts>) is <id>|<keys>|<value>. store.c:store_parse reads BOTH:
field 2 is taken as <ts> only if looks_like_ts accepts it (see <ts>), else it is
<keys>. After at most the first three "|", <value> is the line remainder and may
itself contain "|". A multi-value record (from `add`) is several lines that
share one <id>; it is NOT encoded with extra "|" on one line.

<store>        ::= ( <record> "\n" )*
<record>       ::= <id> "|" ( <ts> "|" )? <keys> "|" <value>
<id>           ::= [:digit:]+            ? positive (> 0), assigned monotonically ?
<ts>           ::= <year> "-" <mm> "-" <dd>
                   ( "T" <hh> ":" <min> ( ":" <ss> "Z"? )? )?
                   ? save time, present in v2/v3 records, absent in v1. looks_like_ts
                     requires at least a full date (YYYY-MM-DD), so a bare year or
                     YYYY-MM stays a KEY. The engine writes UTC to the second with a
                     trailing "Z" (v3); it also accepts the v2 local "...Thh:mm:ss"
                     and the hand-written "YYYY-MM-DD" / "...Thh:mm". A malformed
                     date is read as <keys>, so the line is a dateless v1 record and
                     id/keys/value survive. ?
<hh>           ::= [:digit:]{2}
<min>          ::= [:digit:]{2}
<ss>           ::= [:digit:]{2}
<keys>         ::= <key> ( " " <key> )*  ? one or more space-separated encoded keys ?
<key>          ::= <key_char>+
<key_char>     ::= ? [:graph:] except "|" "/" "\" -- a key is one store line-field
                     AND one safe path component. key.c:key_encode lowercases the
                     source key and maps every other character (space, [:cntrl:],
                     "|", "/", "\") to "_". ?
<value>        ::= <value_char>*         ? may be empty ?
<value_char>   ::= ? any character except "\n" (LF) and "\r" (CR); "|" IS allowed ?

A <value> is the literal resource, conventionally one of the following (these
shapes are descriptive, not enforced: the store keeps any value verbatim):
<value>        ::= <url> | <uri> | <abs_path> | <rel_path> | <secret> | <free_text>
<rel_path>     ::= <blob_ref> | ? any path relative to INDEX ?
<blob_ref>     ::= "blobs/" <timestamp> ( ".txt" | ".aisc" )  ? .txt: `doc`; .aisc: `doc -e` ?

A <secret> is an encrypted value, self-identifying by the "aisc:" prefix (written
by `-e`). INLINE carries the base64 AIS-CR1 image in the value; BLOB references an
encrypted image file (the '@' after the prefix is the discriminator -- base64
never starts with '@'). Treated as opaque <value> everywhere except an
interactive recall at a terminal, which decrypts it:
<secret>       ::= "aisc:" <base64>              ? INLINE: image is the value ?
                 | "aisc:@" <blob_ref>          ? BLOB:   image is blobs/<ts>.aisc ?
<base64>       ::= ? RFC 4648 base64 of the AIS-CR1 file image ?
<timestamp>    ::= <year> "-" <mm> "-" <dd> "-" <hhmmss> ( "-" [:digit:]+ ){0,1}
<year>         ::= [:digit:]{4}
<mm>           ::= [:digit:]{2}
<dd>           ::= [:digit:]{2}
<hhmmss>       ::= [:digit:]{6}


---- rebuildable index files (derivable from <store>, disposable) ------------
INDEX/next_id   -- the next id to assign
<next_id_file> ::= <id> ( "\n" ){0,1}

INDEX/idx/<p>/<key> -- one key's posting list, ascending and duplicate-free
<posting_file> ::= ( <id> "\n" )*        ? strictly ascending, no duplicates ?
<p>            ::= <key_char> ( <key_char> ){0,1}   ? 1-2 char navigable prefix of <key> ?

INDEX/tomb      -- tombstoned (deleted) ids
<tomb_file>    ::= ( <id> "\n" )*

INDEX/multi     -- ids carrying more than one value line
<multi_file>   ::= ( <id> "\n" )*

INDEX/off       -- positional accelerator; line k = byte offset of id k's first
                   store line. Line NUMBER is the id; the stored offset is +1, so
                   "0" means absent.
<off_file>     ::= ( <offset> "\n" )*
<offset>       ::= [:digit:]+            ? offset+1; "0" = no record for this id ?

INDEX/version   -- on-disk format version (file absent = legacy, version 0)
<version_file> ::= [:digit:]+ ( "\n" ){0,1}

INDEX/lock      -- empty; advisory flock target only.


---- real data, not rebuildable ----------------------------------------------
INDEX/blobs/<timestamp>.txt -- a document saved by `doc`
<blob_file>    ::= ? arbitrary text, one or more lines, any bytes ?
(The store references it by <blob_ref>; `find` searches the reference text, not
the blob's contents.)


---- editable batch format (--dump / --import) -------------------------------
`ais --dump` emits <record> lines. `ais --import` reads <import_line> (a <record>
with the leading "<id> |" dropped), skipping blanks and "#" comments. Round-trip:
   ais --dump | sed 's/^[0-9]*|//' | ais --import
Keys are "|"-free (key_encode), so import splits on the FIRST "|"; a value that
itself contains "|" therefore round-trips dump -> import unchanged.
<import_input> ::= ( <import_line> "\n" )*
<import_line>  ::= <comment_line> | "" | <keys> "|" <value>
<comment_line> ::= "#" <value_char>*


============================
NOTES -- verify against the code (the purpose of this document)
============================
1. "|" in <value>: ALLOWED. store_parse takes <value> as the line remainder, so a
   value keeps "|" verbatim (URLs, "a|b" notes).

2. <key> safety is by ENCODING, not rejecting: key.c:key_encode maps to "_" every
   character unsafe for a key (it must be one store line-field AND one path
   component idx/<p>/<key>): see <key_char> above.

3. The conventional <value> forms (url/uri/path/blob_ref) are informative only;
   the engine constrains nothing but "no newline". Only <blob_ref> has a producer
   (`doc`) that guarantees its shape.
